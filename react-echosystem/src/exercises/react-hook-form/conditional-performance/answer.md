# 言語化演習 回答

## Q1. watch() と useWatch の再レンダリング挙動の違い

**回答：**
親コンポーネントで Watch を呼び出した場合は、コンポーネント全体が再レンダリングされるため、そのコンポーネントで呼び出しているコンポーネントも全部再レンダリングされることになってしまいます。一方、useWatch をコンポーネント内で呼び出すことで、再レンダリング範囲をそのコンポーネントのみに閉じることができます。

そのため、親のコンポーネントに影響を与えることはありません。

追記でウォッチを引数なしで呼んだ場合は、全ての項目の変更を検知します。例えば、Name と Age という 2 つの入力フィールドがあった場合、どちらの入力変更も検知して再レンダリングされます。

なので例題でいうと、プランだけ変わった時でも、それともファミリーのネーム変わった時も、運動トリガーにサレンダリングされてしまいます。もし、この項目だけが変わったっていうのを限定したいのであれば、引数でその項目の名前を指定する必要があります。

## Q2. shouldUnregister の使い分け

**回答：**
修団レジスターはUXを優先する場合は基本的にはフォルスにしておいた方が良いかなと思います。つまり値はアマウント時にでも維持されるということを保証した方がいいと思います。

例えば、Q2の例でいうと、家族を選択したときだけ、家族に関わる内容を入力するみたいな切り替えがあるようなフォームでは、一回条件をトゥルーにして値入力してその後フォルスにした場合に、手段レジスターがトゥルーだったら一度入力した内容が破棄されてしまうと思います。ユーザーが間違えてフォルスにしちゃったりっていうチェックを外しちゃったりっていうこともあると思うんですよ。そうしたときにまた入れ直さなきゃいけないっていうストレスを与えてしまうので、操作ミスっちゃったとしても値は維持しておきますっていう風な挙動が手段レジスターをフォルスにしておくことで実現できるかと思います。

なのでUXとUS線したい場合は手段レジスターフォルスにしておく必要があると思います。あると思ってます。ただ、サブミットするときに注意が必要で、本来だったらこの条件のときにこれは送らないのに送っちゃうみたいなケースっていうのが出てきちゃってしまいます。例えば家族に家族ですっていうのにチェックを入れて家族専用の入力フィールドを入れてやっぱりチェック外して登録ボタンを押したときに本来であれば家族専用のフィールドっていうのは送られないはずであるが送られてしまっちゃうっていうような不具合が生じてしまう。つまり画面のUIと送信されるペイロードの値が不整合を起こしてしまうっていう危険性はあるんで、そういった整合性の部分を考えるのであれば手段レジスタはトゥルーにしておいた方が取れるかなと思います。なのでどっちを取るのかっていうのはトレードオフかなと思います。

## Q3. なぜ子では useWatch が推奨されるか

**回答：**
レンダリングの範囲を制限できるかだと思います。ウォッチを使ってそれをコンポーネントに渡すやり方だと、結局親コンポーネントが再レンダリングされると思うんですけど、そうではなく、ユーズウォッチを使うことで、そのコンポーネントだけで監視することで、再レンダリング範囲っていうのはコンポーネントに限定することができるかなと思います。また、Propsっていうのをここに渡す必要がなくなるので、実装もスッキリするんじゃないかなと思ってます。

### 評価コメント

**Q1: 合格** ✔
- `watch()` が呼び出し元コンポーネント全体（＋子）を再レンダリングする点、`useWatch` で範囲を子コンポーネントに閉じられる点、引数なしで全フィールドを購読する点、いずれも正しく理解できている。
- 補足: `watch("plan")` のように引数を指定しても、トリガーされるフィールドが絞られるだけで、再レンダリングされるコンポーネントの範囲は変わらない（呼び出し元全体が再レンダリングされる）。再レンダリング範囲を限定するには `useWatch` を子コンポーネントに切り出す必要がある。

**Q2: 合格** ✔
- `shouldUnregister: true` で値が破棄される挙動、`false` で値が維持される挙動の違いを正しく理解できている。
- UX（操作ミス時の値保持）vs データ整合性（UIと送信データの一致）のトレードオフとして整理できている点が良い。
- 補足: `shouldUnregister: false` を選んだ場合でも、`onSubmit` ハンドラ内で現在の条件に基づき不要フィールドを除外して送信することで、UXと整合性を両立させるパターンも実務では一般的。

**Q3: 合格** ✔
- 親で `watch` → props 経由だと親が再レンダリングされる点、子で `useWatch` を使えば再レンダリングを子に閉じられる点を正しく理解できている。
- props を渡す必要がなくなり実装がシンプルになるというメリットにも触れられている。
- 補足: `useWatch` は `control` を受け取るだけでフォームの値を直接購読できるため、親がフォーム値を中継する必要がなくなる。これにより親は「フォームの構造を定義する」責務に集中でき、各子コンポーネントが「自分に必要な値だけを購読する」という責務分離が自然に実現できる。
