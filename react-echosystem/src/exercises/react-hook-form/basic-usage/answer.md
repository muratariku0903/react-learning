# 言語化演習 回答

## Q1. 非制御がなぜ再レンダリングに効くか

**回答：**
RHFは入力値の状態をステートで管理するのではなく、DOMに任せています。ステートで管理する場合、入力値が変更したタイミングで再レンダリングが実施されるので、基本的に全てのコンポーネント、周囲にあるコンポーネントがレンダリングされますが、React Hook Formでは入力値を変更したところで再レンダリングが走らないため、use stateを使った管理よりもパフォーマンスという部分で優れているかと思います。

HandleSubmit を実行したときに渡ってくるデータは ref から参照してきます。つまりデータは ref に存在しています。
確かに実際にコンソールを出力してみましたが、レンダリングがゼロになるというのは嘘で、レンダリングはされますね。ただ文字を入力するごとにレンダリングされているわけではないく、初回だけレンダリングされているように見えます。初回だけといわれば文字を入力した最初のタイミングでレンダリングされて値を消すとまた再レンダリングされました。

上記のレンダリングのタイミングとしては、購読しているフォームステイトの値が更新されたタイミング、変化があったタイミングにレンダリングされるということでした。今回の実装だと、isDirtyというフォームステイトを購読しており、こちらの変数は入力が変わったか変わってないかっていうのをトリガーしてるみたいです。なので初回に入力するとそれがtrueになり、また戻すとfalseになるという、そういった変化のタイミングでレンダリングが走っていたということでした。

## Q2. Controller が必要になる場面

**回答：**
レジスター経由でREFを注入できないマテリアルUIであったりチャクラUIといったサードパーティー製のコンポーネントでコントローラーが必要になります通常のコンポーネントであればDOMであればREFを注入してそこから値を参照することができるんですけどもサーデパーティーのUIライブラリにレフを渡したところで、そのレフが内部でラップされているインプットに届くかというのは分かりません。なので、そういったレフを注入できないものに関しては、コントローラーを使って別途値を制御して、その制御した値でUIライブラリに反映していくっていうやり方をとります

## Q3. isDirty / isValid の落とし穴

**回答：**
IsDirtyについてですが、落とし穴としては、そもそもisDirtyって何なのかっていうと、入力された値がdefaultValuesと差異があるかないかっていうのを判定して、ある場合はisDirtyがtrueとなります。defaultValuesと比較するため、defaultValuesがundefined、つまり未設定の状態だと、その未設定と入力した値の比較となるので、常にisDirtyがtrueとなってしまうといった落とし穴があります。続いてisValidについてなんですけど、isValidに関してはバリデーションで決めたルールっていうのを入力値が満たしているかどうかっていうフラグになります。こちらの落とし穴としては、基本的にユースケースとして入力値が全部ちゃんとしてたらボタン活性化しますみたいな制御を取り入れたいと思うそういったユースケースが多いかと思うんですけども、実はisValidっていうのはそのままじゃうまく機能しなくて、modeっていう検知タイミングを制御するオプションがあるんですけども、このmodeっていうのがデフォルトでonSubmitになっているので、フォームが提出されたタイミングでisValidというのが判定されるようになっているので、デフォルトのままだと普通にボタンの活性非活性の制御ができないといった落とし穴があります。もしこれをちゃんとした入力されているときだけ活性非活性の制御したいんだったら、modeをonChangeにして入力されたタイミングでisValidかどうかというのを判定してあげるような挙動にする必要があります。またonChangeにした場合は結局毎回毎回レンダリングされることになるので、その場合はパフォーマンス的にも多少懸念点があるところです。
