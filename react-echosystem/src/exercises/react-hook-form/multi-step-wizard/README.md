# 複数ページにまたがるウィザードフォーム

## 目的
複数ステップに分かれたフォーム（いわゆるウィザード）を、
RHF の `FormProvider` と1つの `useForm` インスタンスで構築する方法を理解する。
**ステップごとに別フォームを作る設計** と **1つのフォームを共有する設計** の違いを体感し、
「ステップ単位の部分バリデーション」「戻った時の値の保持」「最終確認画面での読み取り」を扱えるようになる。

## 題材
3ステップの会員登録ウィザード:
1. アカウント情報（email, password, passwordConfirm）
2. プロフィール（氏名、生年月日、性別）
3. 配送先住所（郵便番号、都道府県、市区町村、番地）
4. 確認画面（読み取り専用で全ステップの値を表示し、送信）

## 実装演習

### 要件
`App.tsx` にウィザードのスケルトン（ステップ管理だけ）が用意されています。次を実装してください。

- 各ステップは `components/Step1Account.tsx` `Step2Profile.tsx` `Step3Address.tsx` `Step4Confirm.tsx` に切り出す
- 親の `App.tsx` が `useForm` を1つだけ持ち、`FormProvider` で全ステップに流す
- 各ステップの「次へ」を押したとき、**そのステップのフィールドだけ** を `trigger` でバリデーションし、通った場合のみ次に進む
- 「戻る」では値が消えないこと
- ステップ4（確認画面）では値を読み取り表示し、`submit` ボタンで `console.log`
- パスワード確認は `password === passwordConfirm` を validate ルールで担保

### 制約
- 親の state はステップ番号 (`currentStep`) のみ。フォーム値は `useState` で管理しない
- 子コンポーネントでは `useFormContext` を使う
- `shouldUnregister` の扱いは自分で考えて選択する（理由を `design.md` に書くこと）

### ヒント
- `trigger` には `["email", "password", "passwordConfirm"]` のようにフィールド名配列を渡せる
- 確認画面では `getValues()` を使う or `useWatch` を使うかを検討する（再レンダリング差に注目）
- 各ステップの型は親で1つだけ定義し、`type FormValues` を共有する
