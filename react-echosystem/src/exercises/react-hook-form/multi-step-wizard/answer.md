# 言語化演習 回答

## Q1. (A) 独立 useForm vs (B) FormProvider 共有

**回答：**
最終サブミット時のバリデーション再実行という観点からすると、それぞれのステップをまとめて useForm で管理した方が、各ステップの値を全部バリデーションできると思います。また、仮にそのステップ間の相関チェックみたいなものがあった場合にも、それも連動してバリデーションできるかなと思います。そのため、まとめてバリデーション実行できるという意味では、親で一つだけ useForm を呼ぶという B のやり方に利点があるかなと思います。

また、型の一元管理に関しても、多分 useForm に、各ステップで入力されるべき型を配置しておけばいいはずなので、そうすると、それぞれの入力で型を一元管理できるようになります。今説明した動画です。

今したときの値保持については、A と B で何が違うのかっていうのが、現時点であんまりわかっていません。予想ですが、各ステップで useForm を管理しちゃうと、戻ったときにフォームの値がリセットされちゃうリスクを想定できます。そのため、親で一つだけ useForm を読んだ方が値の維持しやすいんじゃないかなと個人的には思います。

そのため、整理すると B のやり方が好ましいかなって思っています。ただ、ステップごとに useForm、つまり A の利点としては、ステップごとに保存したりっていう処理は普通にあると思うんですよね。今回だと会員登録のウィザードなんで、ステップ 1 とステップ 2 を途中で保存したりみたいな要件の場合は、useForm をステップごとに配置すると、そこで保存処理ができるっていう要件によるんじゃないかなって個人的には思います。

値保持とバリデーションについて補足します。まず A の場合の値保持についてなんですが、ステップを前後するたびに useForm というのは消失します。useForm を使っているコンポーネントがマウント・アンマウントされるごとに値が初期化されるため、つど大元の状態を用意して、そこを繰り返ごとに更新していく仕組みが必要になります。

なので、useForm、RHF とはまた別に値保持を管理するような仕組みを検討する必要が出てきます。また同じく A のバリデーションについてなんですけども、こちらも最終ページに到達した際には MyStep のフォームの値っていうのがアンマウントされている状態になります。なので、バリデートする値は大元のステートから引っ張ってくる必要があって、その値に対してバリデーションをする必要があります。そのため、RHF と特有のバリデーションの機構を生かすことができず、自前で合成した状態に対してバリデーションを考える必要が出てきます。

## Q2. trigger() 引数なし vs フィールド指定

**回答：**
まず段階的にバリデーションする場合は、引数を指定してトリガー関数を実行しておくのが良いと思います。トリガー関数で引数を指定しない場合だと、全てのフォーム、全ての入力値をバリデーションしてしまうので、まだそもそも入力してない後続のステップの値をバリデーションしてしまうため、必ずエラーになってしまうので、特定のステップの値だけバリデーションする必要が出てきます。

そのために、トリガーに引き継いで、どの入力値をチェックするのかというのを示す必要があります。

また引数なしでバリデーションしてしまうと、後続のステップですでにエラーとなっている状態で表示されてしまう可能性があるため、ユーザーから「なんでまだ入力してないのに怒られてるの？」みたいな反感を買う可能性があるので、UX的にも引数を指定した方が良いと思います。

## Q3. ステップ間でフィールド値が消える問題

**回答：**
Unmountされたフィールドのデータは消失してしまいますが、shouldUnregisterというオプションが存在し、こちらをfalseにすることでUnregisterされない、つまりずっと残り続ける状態にできます。そのため、多段フェーズになるフォームに関してはshouldRegisterをfalseに設定して、後続のフィールドやアンマウントされたフィールドを維持するようにすることが可能です。

ただし、こうすることによって画面では表示されていないフィールドが維持され続けるため、その値が何かしらの処理に使われ続ける可能性がある点に注意が必要です。例えば、何らかのフィールドを有効化した場合に別のフィールドが出現したり、そのフィールドが無効化されたときに非表示になったりといった相関的な動きがある場合、shouldUnregisterはtrueにしておいた方が安全な動作になると考えます。

手段レジスタをフォルスにすることによって、RFHの内部ストアがデータを保持し、再度マウントされたときにDOMとRFHの内部ストアを紐づけます。

RFH Ver.7の場合、shouldUnregisterのデフォルト値はfalseになります。なので何も指定しない場合はフィールドの値は残り続ける性質です。
