# 言語化演習 回答

## Q1. key に index を使うと何が起きるか

**回答：**
React自体がそのキーをもとに同一のレンダリング対象かどうかっていうのを判断すると思っていて、ちょっと言葉で表現するのがちょっと難しいんですけど、おそらくその同一判断する処理がインデックスをキーとすることで狂ってしまって、結果として全然違う要素に入力値が入ってしまうみたいなことが起こっちゃうんじゃないかなって思ってます。例えば3つの要素があって上から0,1,2というインデックスが振られ、それをキーに設定した場合、例えば真ん中のインデックス1の要素を削除した場合、配列の要素は2つになって、1番末尾の3番目の要素が2番目に昇格するじゃないですか。そうするとインデックスをキーとしているので、React的には前の多分内部でファイバで管理していると思うんですけど、1,2,3だった時の2の要素と現在1,2ってなっている2番目、その前の3番目の要素を比較すると思うんですよ。比較した結果、おそらく再レンダリングが走るんですよ。走る必要ないんですよ。3番目って何も編集してないから、ただ上に昇格しただけだから、2番目が削除されたことによってっていうような不必要なレンダリングが生じるんじゃないかなって個人的には思ってます。また追加の例を言うと、また0,1,2っていうのがあって、0番目を消したとき、2、いや、なんでもないです。おそらく、つまり、要するに、React的にはキーを元に同一コンポーネントかっていうのを判断してるんで、そのキーが配列の要素の増減によって変わってしまうと、同一判定が適切に行われなくなってしまうんですよ。

また RHF にまつわる懸念点でいうと、中間要素を削除した場合に表示がずれてしまうという事態が発生します。まず RHF の場合は値の管理場所は DOM になります。React のステートではないものになるので、そもそもの差分検知の仕組みというのが存在しないことになります。もう少し具体的に言うと、register という登録用の関数を DOM に流し込んでいるんですが、そこには値というものが含まれていません。いわゆる入力値ですね。name であったり、onChange といったハンドラがスプレッドされているだけなので、つまりこの値が変更ない場合に、RHF では key が紐づくものをそのまま表示することになります。なので例えば 3 つの要素があったときに 2 つ目の要素を削除した場合は、3 つ目の要素の index が 2 つ目に食い上げられてしまうんですが、その場合はちゃんと 3 つ目の要素が表示されるべきなんですが、そういったことには RHF は検知できないので、削除されたはずの 2 つ目の要素で 3 つ目の要素が上書きされてしまうということになります。上書きというか、あくまで key が 1 のものの入力値が表示されるという扱いになるので、結果として表示がずれてしまうといったことが起こります。

## Q2. 子コンポーネントに useFieldArray を切り出す設計の利害

**回答：**
コンポーネントで useFieldArray を切り出す場合、useFormContext というフックを使って、useForm の返り値を監視することになると思います。これによって、親からプロップスをバケツリレーする必要がなくなるので、コンポーネントを呼び出している箇所がすっきりするというところがあると思います。また、レンダリングの範囲を限定的にできるんじゃないかなと思っています。もし全てを親で管理する場合、コンポーネントのリストを削除したり追加したりというのがフィールドの変更によって親に検知されてしまうので、親全体のコンポーネントが再レンダリングされるんじゃないかなって思ってるんですけど、コンポーネントで useFieldArray を呼び出す場合は、そのコンポーネントのみがレンダリング対象になるんじゃないかなって個人的には思ってます。デメリットとしては、状態を管理しているパーツが分散するので、どこで何をやっているのかが把握しにくいというのはあると思います。これは親で管理するパターンの方がそこを見ればわかるので、状態把握という点ではデメリットかなと思います。

上記の説明にFBがありました。親でuseFieldArrayを呼び出すと、毎回親のコンポーネント全体が再レンダリングされると言ったんですけど、厳密に言うと、入力値を変更したときは再レンダリングされず、行の追加・削除、つまり配列の構造を変えるアクションをしたときに再レンダリングが走ります。この場合に各コンポーネントをReact.memoでメモ化しておくことで、関係ないところの再レンダリングを抑えることができるという認識です。また、コンポーネントに切り出すメリットとして、レンダリング範囲を限定するというところをもう少し話すと、useWatchっていうのを内部で呼び出すことで、観測範囲を狭めることができます。これをもし親でuseWatchをしてしまった場合、あらゆる変更によってウォッチがトリガーされてコンポーネントが全行評価されるということが発生するんですが、そのウォッチを子コンポーネントに限定してあげることで、その再レンダリングっていうのがその子コンポーネントだけに抑えることができるというメリットがあると思います。

## Q3. 動的 name のバグパターン

**回答：**
典型的なパターンとしては、キーとネームにインデックスを割り振っているパターンですね。さらにそれが入れ子構造になっていると、DOMで保持している値とRHFの内部ストアで管理する値がずれてしまうことで、バグにつながることが考えられます。

例えば、具体的に1、2、3という3つの要素があり、それをArrayFieldsで管理していて、それぞれのリスト要素に対してキーとネームにインデックスを入れていた場合、先頭の要素を削除したとします。そうするとRHFの内部ストアでは名前が調整されて繰り上げられますが、DOM側ではキーを使ってコンポーネントの同一判定を行っているため、2個目の要素が本来最初に表示されるべきなのに、削除されたはずの1個目の要素がDOM上に表示されてしまいます。その結果、DOMの値とRHFで管理している値がずれることになります。

この状態でsubmit処理を行い、リクエストにデータを付与しようとすると、RHFの内部ストアから値を参照するため、ユーザーが「a」を送信しているはずなのに、内部では「b」を送信してしまうような食い違いが発生します。

補足として、なぜ一番目の要素が削除されたのにも関わらず、DOMとして残り続けるのかというのは、非制御コンポーネントの特有の仕組みであり、入力のバリューというものを差分検知できないからですね。

RHFで差分として検出されるのはバリューではなくて、ネームであったりオンチェンジであったりオンバリューといった値になるレジスターによってスプレッドされたものになります。今回、先頭の要素を削除したとして、バリューは変わるんですけども、その検知がされないということになって、制御コンポーネント的には、ネームとかが変わらないのであれば、それは同じ要素であると判定されるので、そのまま一番目の要素が使い回されるというような挙動になります。
