# 言語化演習 回答

## Q1. zodResolver による変化（型安全性・一元管理の観点）

**回答：**
バリデーションロジックの一元管理という点からすると、DODOで定義したスキーマにロジックを集約することで、入力値の検証であったり、あるいはリクエストのAPIでの検証をするときに、同じようなバリデーションを流用できる利点があります。

もしこれをRHFでやるとすると、おそらくレジスタ関数の引数としてバリデーションの定義を追記しなければいけないので、他のコンポーネントに対して流用することが難しいと考えられます。

また、型の安全性という点でいうと、これもバリデーションロジックの一元管理に関わってくる部分ですが、スキーマから型を作れるため、そのスキーマとずれていない型が生成できます。そのため、入力値と矛盾しない型を作れ、かつその型をさまざまな場面で利用できるため、間違った型を使うことがなくなり、型安全性が優れていると考えます。

## Q2. z.infer による型導出のメリット

**回答：**
こちらもQ1で説明した通り、バリデーションと型が乖離しないという安全性が得られるかなと思います。もしZodのインファで型を導出しなかった場合、独立して型を定義するかと思います。その場合はそのバリデーションと一致しているかどうかっていうチェックが手動で入るんで、人為的なミスであったりズレみたいなのが発生しえるが、Zodのインファを使うことで隙間を神としてそこから矛盾のない型を自動生成できるというところで、矛盾が生じないというメリットがあります。

インファで推測した場合、オプショナルであったりデフォルトトランスフォームなどを適用した場合は、それを加味した型が推測されます。例えばオプショナルを付与した場合はUndefinedを許容するようになりますし、デフォルトを設定した場合は必ず何かしらの文字列や数字がある方がオプショナルにはならなくなる方が推論されます。なのでこうした変化があったとしてもそれと同期した方が自動的に連動するのでメンテナンスによる不具合などが起きにくい構成を作ることができます。

## Q3. Zod のエラーメッセージと formState.errors の関係

**回答：**
Zodでは、スキーマで定義されるルールや、そのルールに違反した際のエラーメッセージを管理し、それがフォームステイトのエラーズに変換される仕組みになっています。例えば、スキーマでストリングを付与した場合、formErrorsにはまずどの項目なのかがパスに格納され、エラーのコードとエラーメッセージがセットで用意される形になります。そのため、Zodを使ったからといってreact-hook-formのエラーフォーマットが崩れるわけではなく、あくまで同じ構成にコンバートされる形になります。

ZODのエラーをRHFのエラーの形式に変換しているのがResolverになります。\n\nなので、ZODとRHFをつなぐパッケージが別途必要になります。ZOD Resolverというものですね。
