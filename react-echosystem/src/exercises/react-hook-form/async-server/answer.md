# 言語化演習 回答

## Q1. Zodの refine による非同期バリデーションの実行タイミングとmodeオプションの関係

**回答：**
バリデーションの実行タイミングは、Zod Resolver の設定や React Hook Form のバリデーションモードの設定によって主に3つあります。サブミットされた時、onBlur（フォーカスが外れた時）、onChange（変更の度に）の3つのタイミングで、基本的にバリデーションが実行されます。あるいは、特定のタイミングで実行させたい場合は、トリガーなどを使用します。

Zod の refine による複雑なバリデーションの実行タイミングとモードオプションの関係についてですが、例えばモードが onChange だった場合、入力が変更されるたびに refine が実行されます。これは、例えば refine 内で API 通信を行っている場合、入力のたびに API が呼ばれることになり、負荷が大きくなる可能性があります。その結果、入力が遅くなったり、UX の低下を招くリスクがあります。

そのため、基本的には onBlur や submit のタイミングで refine による非同期バリデーションを実行するのが、UX 的に好ましいと考えられます。

## Q2. setError と zodResolver が設定するエラーの違い

**回答：**
両者いずれもRHFで解釈できるErrorsというオブジェクトに変換するためのものです。setErrorはプログラムでフォームのパスに対して、特定のエラーメッセージを直接設定するものです。Zod.Resolverは、内部でZodがバリデーションしたエラーをRHFが解釈できるエラーに変換するものです。

それぞれユースケースが異なります。基本的にZod Resolverは、入力値の検証や入力値の相関検証、つまりクライアント側に閉じた検証に使われることがメインです。一方、setErrorは入力値が正しいかどうかという意味ではなく、サーバー側との矛盾、競合や権限などのエラーをRHF側に統合するために設定します。

したがって、役割が異なります。Zod Resolverはユーザーが入力した値を処理し、setErrorはサーバー側が返してきたエラーの値を設定するものです。

Zod.resolver が設定したエラーは、次のバリデーションが走ったときにまた新しく Zod によって評価され再設定されることになります。例えば、値が変わっていた場合にバリデーションしてその結果問題なかった場合は、エラーが消えます。

一方、setError で手動で設定したエラーに関しては、クリアリング処理をしない限りは基本的にはずっとそこに残り続けることになります。ただし、setError で出しているエラーと Zod resolver で出しているエラーの箇所が重複している場合は、Zod resolver によってエラーが上書きされる可能性もあります。したがって、setError で出しているエラーがずっと残るかと言われると、そうではないケースも存在します。

## Q3. refine による非同期バリデーション vs onSubmit 内の setError

**回答：**
Zod Resolverのメリットとしては、バリデーションのルールを一箇所に集約することができる点です。例えば、setErrorなどを使ってバリデーションロジックを書くと、いつバリデーションを行うのか、どのようなメッセージを出すのかといった内容がロジックの中に混ざってしまい、確認すべき場所が増える傾向があります。Zod Resolverを使うことで、エラー周りの設定を一箇所のファイルに集約できるため、管理がしやすくなります。

一方でデメリットとしては、エラーバリデーションのトリガーとしてReact Hook Formで用意されているものしか使えないという点が挙げられます。onSubmitやonChangeといった決められたフェーズでのみバリデーションが実行できるため、柔軟性に欠けると感じる場合があります。例えば、APIの実行結果によってエラーメッセージを出したいといった要件がある場合、Zod Resolverではそのトリガーがどこにあるのかが明確でなく、柔軟性が発揮しにくいです。

setErrorに関しては逆のアプローチが取れます。メリットとしては、プログラマティックにsetErrorを実行することで、開発者が好きなタイミングでエラーを設定できる柔軟性がある点です。ただし、その分ロジックが分散しやすくなるという点は注意が必要です。
