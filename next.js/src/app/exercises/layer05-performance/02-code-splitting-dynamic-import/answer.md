# 演習5-2: Code Splitting & Dynamic Import — 言語化演習回答

## Q1: Code Splitting とは何か

コードスプリッティングとは、一括でコードを読み込むのではなく、分割してコードを必要なときに読み込むという仕組みです。これによって初回の画面表示の速度が上がるので、それによるユーザー体験の向上が期待できます。また、不必要なJSのロードを抑えることができるので、ネットワークコストも下げることができます。もし分割ロードをせずに一括で取ってきてしまう場合は、実際に活用されていないJSやJavaScriptのソースなどを読み込んでしまうため、初回の画面表示が遅延してしまったり、ネットワーク帯域を圧迫してしまうことにもつながるかと思います。

ユーザー体験の観点から言うと、コードスプリッティングをすることで、初回の画面表示であったり、その後のユーザー操作がもたつくといったことを解消することができます。必要な分だけのJSをロードすることによって、ブラウザが早々ロードする時間が短縮できますし、その後のJSの実行開始も向上することで、サイトがすぐにアクティブ化されるためです。

ネットワークコストの観点から見ても、コードスプリッティングをすることで、必要な部分だけの通信に抑えることができるので、ネットワークコストを一定削減することが可能になると思います。

例えばECサイトを想定して具体例を挙げると、例えばユーザーが商品一覧ページを見ているとします。ここでコードスプリッティングを使わずに一括で早々ロードするようになっていた場合、例えばチャートを描画するためのライブラリなども読み込まれることになります。チャートのライブラリなどはコードの行数なども大きいと予想できるため、初回のロード時間であったり、そのJavaScriptがブラウザの中でロードされて実行されるという時間も考慮すると、ユーザーにとってサイトがすぐにアクティブにならないので、UXは低下するかなと思います。ネットワークの通信コストも大きくなるのかなと予想できます。ここでコードスプリッティングを取り入れることによって、商品一覧をユーザーが見ているので、チャートのライブラリなどは不要とみなし、必要に応じて後からインポートすることができるので、一旦はユーザーがサイトをすぐに操作することが可能になり、UXの向上が期待できます。

## Q2: Next.js の自動 Code Splitting

Next.js はデフォルトでルート単位でコードスプリッティングを行ってくれます。例えばページAとページBがあったときは、ページAだけで必要なソースとページBだけで必要なソースっていうのはそれぞれ別チャンクに分かれます。ページAにアクセスしたときはページA専用のチャンクだけが、チャンクと共通のチャンクがクライアントからロードされることになります。
また同じページの内でもそれぞれ重たい処理などはやはり別チャンクに分けている必要があると思います。例えばまっすぐにアクティブ化する必要のないチャート、ライブラリなどを使うケースですね。こうした場合はダイナミックインポートというNext.jsが用意しているものを使うことによって別チャンクに分けることができます。そうすることによって、例えばページAにチャートを表示するパーツがあったとしても、そこのパーツに関してはすぐにロードされるものではなく、必要になったときにロードされるようになるので、結果として、ページAのTTIというのが短縮されるので、表示速度も速くなるし、UXの向上が期待できるかなと思います。

2. **もう一つ重要な `dynamic()` のユースケースがあります。** ページ内の「初回に不要な重いコンポーネント」は押さえられていますが、**ブラウザ専用のAPI（`window`や`document`）を使うコンポーネント**の場合はどうでしょうか？ SSRのときにサーバー側でそれらのAPIは存在しませんよね。これは Q3 で扱うテーマなので、そこで意識してみてください。

3. **自動分割の補足:** ルート単位だけでなく、Next.jsは**`layout.tsx`や`loading.tsx`などの特殊ファイルも別チャンクに分割**します。つまり、共有レイアウトのコードはそのレイアウトを使うすべてのページで共有され、ページ遷移時に再ロードされません。これもNext.jsが自動でやってくれる最適化の一つです。

## Q3: `dynamic()` と React の `lazy()` の違い

Next DynamicはReact Lazyと違って、SSRでレンダリングさせるかさせないかの指定オプションがサポートされています。React Lazyの場合は、SSRの挙動までは細かく定義できないですけども、Next Dynamicの場合は、SSR: falseと指定することで、そのコンポーネントのレンダリングをサーバー側ではなく、確実にクライアント側で実施することを強制できます。これのおかげで、例えばwindow依存しているコンポーネントを確実にサーバーでレンダリングさせないといった安全なレンダリング戦略を取ることができます。また、ローディングの表示についても、React Lazyの場合はサスペンスでラップする必要がありますが、Next Dynamicの場合は、ローディングプロパティにローディング中に表示されるコンポーネントを指定する必要があります。

## Q4: Prefetch の仕組み

Prefetch はユーザーが次に遷移しそうなページで必要とされるソースを事前に回収する仕組みになります。これによってユーザーがリンクをクリックしたときに事前に回収したソースを元に画面を表示するので即座に画面が表示されて UX を向上することができます。もう少し詳しく説明すると、この Next.js の Link コンポーネントがビューポートに表示された直後ぐらいのタイミングで、裏でそのリンクのページ先で必要とされるチャンクをあらかじめロードしてくれます。これによってリンクが実際にクリックされたときに取得したソースを使って画面を処理することができるので、遷移した直後にリソースをロードすることを極力抑えることができます。

静的ルートと動的ルートで若干挙動が異なっていて、静的ルートの場合は CDN キャッシュから Prefetch してきますので、Prefetch した内容をそのまま遷移先の画面で表示するだけなので、遷移先の画面で再度リクエストをするといったことは基本的には起こらないと思います。そのため即座にページが表示されるものになります。動的ルートの場合は、Ninker であったり Cache No Store といったページで表示するコンテンツの鮮度っていうのが大事になってくるものになりますので、基本的にはリンクをクリックしたときにデータを取ってきてそれを表示するみたいな動きになるので、Prefetch したものだけで遷移先の画面を表示することは難しいです。なので遷移した後でも何回かリクエストして画面のコンテンツを取得する動作をすることになります。

ただ動的ルートでも Prefetch のメリットはあって、その遷移で鮮度がそこまで大事じゃないもの、例えば画面のレイアウトであったりデザインだったり画像だったりっていう点に関しては、Prefetch でチャンクごとに分かれているのでそれを取ってきて遷移後の画面で適用するといった動きができるので、そこに関しては静的ルートと同じように画面表示の速度の改善が期待できるかなと思います。

`prefetch={false}` を使うべきケースとしては、先ほど言ったように、基本的にリンクをクリックしたタイミングでデータを取りたいときですね。鮮度っていうのをすごく大事にしている画面のコンテンツなどは、Prefetch を false にしておいて、都度取得してくるという挙動をしておいたほうが良いかと思います。
