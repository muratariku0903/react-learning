### 実装レビューコメント

**実装1・実装2: 合格です。** 要件を満たしています。

- `HeavyComponent` — `heavyFunc()` で10万件のループを回して「重さ」を演出し、`dynamic()` + `loading` で遅延読み込みできています。
- `BrowserOnlyComponent` — `window.innerWidth`/`innerHeight` を使い、`ssr: false` で正しく読み込めています。

**実装の指摘（2点）:**

1. **`prefetch={true}` のリンクがコメントアウトされています**（page.tsx 99〜108行目）。3パターンの比較が要件なので、コメントアウトを解除して3つとも表示されるようにしてください。

2. **リンクのテキストと実際の遷移先が一致していません。** 3つのリンクすべてが同じ `01-image-font-optimization` を指していますが、テキストでは「演習5-1へ」「演習4-1へ」「演習3-2へ」と別々の画面名になっています。混乱を避けるため、テキストを揃えるか、遷移先をそれぞれ別のページにするか、どちらかに統一しましょう。

**notes.md の観察について:**

- 実装1・実装2の観察は的確です。
- 実装3について一つ質問です: **デフォルトと `prefetch={true}` が「同じ挙動」と書いていますが、本当に完全に同じでしょうか？** 言語化演習Q4で学んだ「静的ルートと動的ルートでPrefetchの範囲が異なる」ことを踏まえると、デフォルトと明示的trueで取得される範囲に違いがある可能性があります。確認してみてください。
Prefetchの取得先を動的ルートにして、デフォルトとprefetch-trueの違いを調べてみたところ、デフォルトの場合は、遷移前に遷移先のRSCフェッチリクエストが実行されており、遷移後にSSRのfetchリクエストが実施されていました。
対して、Prefetch: trueを強制した場合は、事前に遷移先のRSCのfetchリクエストが実施され、該当ページに遷移した後はリクエストは実施されませんでした。

### 追加検証レビューコメント

**素晴らしい検証です。** 動的ルートに変更して比較したのは正しいアプローチです。観察結果も的確で、まさにデフォルトと `prefetch={true}` の核心的な違いを捉えています。

整理すると:
- **デフォルト（動的ルート）:** レイアウトや `loading.tsx` など**共有部分のみ**を事前取得。ページ固有のデータは遷移後に取得される（= 部分的Prefetch）
- **`prefetch={true}`（動的ルート）:** **フルルートのRSC Payload**を事前取得。遷移後の追加リクエストが不要になる

つまりデフォルトは「鮮度を優先して最小限だけ事前取得」、`prefetch={true}` は「速度を優先してすべて事前取得」というトレードオフです。静的ルートではこの違いが出にくかったのも、静的ルートはそもそもデータの鮮度を気にする必要がないためです。
