## 実装レビュー

### 全体評価
設計メモ（design.md）の要件をほぼ網羅しており、Server Actions の基本パターンを正しく実装できています。Server/Client Component の使い分け、`useActionState`・`useFormStatus`・`useOptimistic` の使い方も適切です。

### 良い点
- **Server/Client の境界が適切**: `page.tsx`（Server Component）でデータ取得 → Client Component に props で渡す設計が正しい
- **`SubmitButton` の分離が正しい**: `useFormStatus` は `<form>` の子コンポーネントでしか動作しないため、別コンポーネントに切り出す必要がある。これを正しく理解して実装している
- **zod バリデーション**: Server Action 内でサーバー側バリデーションを実装している（Q5 で学んだ通り）
- **`useOptimistic` + `useTransition`**: チェックボックスの楽観的更新が正しいパターンで実装されている

### 指摘事項

#### 1. 削除機能が未実装（設計との不整合）
`_actions/todo.ts` に `deleteTodo` は定義されているが、`TodoItem.tsx` に削除ボタンがない。design.md では「削除機能（`deleteTodo`）」が要件に含まれている。削除ボタンを追加してください。

#### 2. `AddTodoForm` でエラー時もタイトルがクリアされる
```tsx
// 現在の実装（AddTodoForm.tsx:12-13）
setTitle("");  // ← バリデーション結果に関係なく常にクリアされる
const result = await addTodo(formData);
```
バリデーションエラーの場合、ユーザーは入力内容を修正したいはず。成功時のみクリアするように修正を検討してください。

#### 3. `useOptimistic` の連続クリック問題（理解確認の質問）
`TodoItem.tsx:17` で `setCompleted(!todo.completed)` としていますが、`todo.completed` はサーバーから渡された props の値です。ユーザーがサーバー応答前に素早く2回チェックボックスをクリックした場合、どうなるでしょうか？ `completed`（楽観的な値）ではなく `todo.completed`（props の値）を反転しているため、2回目のクリックでも同じ値が設定されてしまう可能性があります。この点について考えてみてください。
