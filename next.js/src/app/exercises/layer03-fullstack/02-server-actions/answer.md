## Q1: Server Actions の仕組み

**HTTP レベルで何が起きているか:**
HTTPレベルではPOSTリクエストが送信されています。関数を実行しているように見えるんですけど、内部的にはプロキシ関数というのが実行されていて、そいつがPOSTリクエストを呼んでいます。


**なぜ関数を直接呼び出すように書けるのか:**
仕組みを説明すると、アプリケーションがビルドされると、サーバーアクションはクライアント側とサーバー側にそれぞれ分離されます。クライアント側ではプロキシ関数に書き換わり、そこにはサーバーアクションのIDを引数とする関数になります。サーバー側では実際に処理とIDのマッピングが保存されています。一旦そういう2つのクライアント側とサーバー側に分離されて、次に、このような仕組みを作成すると、クライアント側で関数を実行すると、内部的にはクライアント側の関数、プロキシ関数というのが実行されて、そこのIDと紐づいている処理というのが最終的にサーバー側で実行されるというような仕組みになっています。こういった仕組みが隠蔽されることによって、あたかもサーバー側の関数を実行しているように見せかけているわけです。


**引数・戻り値の制約（シリアライズの観点）:**
引数であったり戻り値などは純粋な値、プリミティブな値にする必要があります。関数であったりクラスなどは渡したり返却したりすることができません。関数であったりクラスといったものをブラウザで同等の環境を構築することが難しいからです。例えばサーバーで定義した関数というのは、その処理だけではなくて、その関数が定義されたコンテキストも含みます。それらをJSONにしてブラウザで同等の環境を再現することは難しいです。クラスも同じです。

> **レビューコメント:**
>
> **HTTP レベル・プロキシ関数の仕組み** — 正確に理解できています。ビルド時にクライアント側はプロキシ関数に置き換わり、サーバー側にはIDと実処理のマッピングが残る、という説明は的確です。
>
> **シリアライズの制約について追加質問:**
> 「プリミティブな値にする必要がある」と書いていますが、これは少し範囲が狭すぎるかもしれません。例えば、`{ name: "test", tags: ["a", "b"] }` のようなプレーンなオブジェクトや配列は渡せるでしょうか？ また、Server Actions の引数として `FormData` を受け取れるのはなぜでしょうか？
>
> 「シリアライズ可能な値」とは具体的にどこまでの範囲を含むか、もう少し考えてみてください。React の flight protocol がサポートする型（Date, Map, Set なども含む）という観点で整理すると、より正確になります。

プリミティブな値というのは少し範囲が狭すぎましたね。要するに、JSON に変換してそれを元通りに誤差なく復元できる形式のデータであれば問題ありません。配列であったりオブジェクトであったり、数値とか文字列とか、それらはサーバーだろうがクライアントだろうが復元可能なので、そういうのを多分 RSC ペイロードの flight プロトコルがサポートする方になるのかなと思います。

## Q2: form の action 属性と Server Actions

**従来の onSubmit + fetch パターンとの違い:**
従来のonSubmitプラスfetchでは、イベントのデフォルト制御を無効にしたり、fetchを使って自前でリクエストを構築したり、そのfetchをuseStateなどを使って状態管理をしたり、エラー処理などをtry/catchでハンドリングしたりといった、リクエストとその周辺の制御を自前で構築する必要がありました。こうした手間のある制御をある程度省略してくれたものがサーバーアクションズとなっていて、useActionStateを組み合わせることで、単純に関数を実行するだけで、ローディング状態だったりエラーだったりを制御できるようになったかなと思います。また、サーバーアクションズは内部的にstartTransitionを使うため、サーバーアクションズが完了した後のレンダリングに関しては、コミットっていうのが遅延されるようになります。この性質の恩恵により、裏でAPIを呼び出している間、ユーザーの制御をブロックするということがなくなるというメリットもあります。


**JavaScript が無効な環境でも動作するか:**
JavaScriptが無効な環境でも、Server Actionsは動作します。JavaScriptが無効な場合は、フォームのもともと搭載されている機能を利用してリクエストを送信するようになります。


**useActionState を使うメリット:**
サーバーアクションズの処理の状態や制御との相性が良く、具体的に言うと、サーバーアクションズが処理中であったり、処理した後の状態などを受け取れるような仕組みが用意されているので、そうした仕組みを利用することで、例えばローディング状態であったり、更新した値で画面の内容を更新するといったことが簡単にできるようになります。

> **レビューコメント:**
>
> **従来パターンとの違い** — 実装コストの低さに触れている点は良いです。ただ、もう少し具体的な違いを考えてみてください:
> - 従来パターンでは `preventDefault()` → `fetch()` → レスポンス処理 → state 更新 → エラーハンドリング… と開発者が手続き的に書く必要がありました。Server Actions では、このうちどの部分が不要になりますか？
> - もう一つ重要な違いとして、**React のトランジションとの統合**があります。Server Actions は内部的に `startTransition` でラップされるため、UI の応答性にどういう影響がありますか？
>
> **JS 無効環境での動作** — 正しいです。`<form action={serverAction}>` が HTML の標準的なフォーム送信にフォールバックするという理解は的確です。これを **Progressive Enhancement（プログレッシブエンハンスメント）** と呼びます。では逆に、従来の `onSubmit + fetch` パターンは JS 無効環境で動作するでしょうか？ なぜ差が生まれるのか考えてみてください。
>
> **useActionState のメリット** — ローディング状態の管理ができる点は合っています。補足として、`useActionState` が返す `state` にはサーバーからの**戻り値**（バリデーションエラーなど）が入る点も重要です。従来パターンだと `try/catch` と `useState` を自前で組み合わせる必要がありましたが、`useActionState` ではそれが一つの仕組みに統合されています。この点も回答に加えるとより完全になります。

## Q3: Optimistic UI

**楽観的更新とは何か・なぜ UX が向上するか:**
一旦画面の内容を更新し、その裏で実際にデータベースの内容を更新する。何かあったときにうまくいかなかったときは、その元の値で画面の状態をリセットするというのが楽観的更新になります。こうすることによって、ユーザーの操作によってすぐに画面が更新されるので、UXが向上します。楽観的更新をしなかった場合は、裏側の処理が完了するまでにユーザーは待機することになってしまいます。 なので、こうした待機時間を極力なくして、何かあったときにUIをリセットするという仕組みがあることで、UXの向上と矛盾、データの整合性というものを担保するというのが楽観的更新のメリットかなと思います。


**サーバー処理が失敗した場合の挙動:**
サーバーの処理が失敗した場合は、元々の既存の値にリセットされます。


**楽観的更新が向いている操作・向いていない操作:**
向いていない操作に関しては、サーバーとクライアントで常に同期している必要があるセンシティブなデータは、楽観的更新をしない方がいいんじゃないかなと思います。例えば証券サイトや銀行のサイトなど、お金が関わるサイト、自分の残高を表示したりする場合は、常にサーバーと同じ値を見せておく必要があると思います。その分の作業を進めて動作を返すことで、処理が失敗して残高が減ったり増えたりしたら、ユーザーにとって大きなストレスになるためです。こうしたセンシティブなデータは、楽観的更新は向いていないのかなと思います。

また、楽観的更新をしたことによって更新をトリガーとして、さらなる画面上の動きが連動するケースの場合は、連動した動きもリセットする必要が出てきてしまうし、場合によってはリセットできなくなってしまったり、そういった副作用みたいなのがあるので、楽観的更新によって絡むのであれば、なるべくそういうのは避けた方がいいんじゃないかなと思います。逆にそれ以外でいうと、楽観的更新を基本使ってUXを上げるっていう方向で良いかなと考えています。
逆にいいねであったり、ブックマークであったり、to doの完了チェックなど、そこだけで完結するようなもの、それをトリガーにして新たな画面遷移が可能になるとか、そういう副作用を持たないものに関しては、失敗することのダメージっていうのは全然大きくないので、そういう場合はロールバックすればいいと思うので、そういう場合は楽観的更新を適用した方が良いと思います。

> **レビューコメント:**
>
> **楽観的更新の定義・UX向上** — 「先にUIを更新し、失敗時にリセットする」という核心を正確に捉えています。待機時間をなくしつつデータ整合性も担保するという整理も良いです。合格です。
>
> **失敗時の挙動** — 正しいですが、もう一歩踏み込んでみてください。`useOptimistic` の場合、具体的にどういう仕組みで「元の値にリセット」されるのでしょうか？ ヒント: `useOptimistic` に渡す第1引数（サーバーから取得した実際の値）が、Server Action 完了後の再レンダリングで更新されることがポイントです。つまり、開発者が明示的にロールバック処理を書くのではなく、React のデータフローの中で自然にリセットされます。この点を理解しておくと実装時に役立ちます。
>
> **向き不向き** — 金融系のセンシティブなデータや、副作用が連鎖するケースが向いていないという指摘は的確です。「向いている操作」についても具体例を挙げると回答がより完全になります。例えば「いいね」「ブックマーク」「Todo の完了チェック」など、失敗してもロールバックのインパクトが小さい操作はどうでしょうか？

## Q4: Server Actions と Route Handlers の比較

**CRUD 操作を Server Actions で実装する場合に何が変わるか:**
サーバーアクションズでCRUD操作を実装する場合は、比較的ルートハンドラーズよりは実装コストは低くなると思います。Route Handlersの場合はエンドポイントを定義してそれを呼び出すコンポーネントでフェッチを実行する必要がありますので、リクエストを自前で構築したり、そのAPIのレスポンスの結果のエラーハンドリングだったり、ローディングなども自分でやる必要が出てくるかなと思います。対してサーバーアクションズは単純に関数を定義してそれをクライアントコンポーネントで呼び出せばそれだけで勝手にリクエストを実行してくれますし、ユーザーアクションステートを使うことでそのリクエストの状態っていうのを簡潔に管理ができると思うので、リクエストとその制御っていうのを比較的Route Handlersよりも手軽に実装できるかと思います。


**Route Handlers で実装すべきケース vs Server Actions で実装すべきケース:**
こちらは様々な観点があるかなと思いますまず一つ目の観点としてはその処理を誰が呼ぶのかというのがありますウェブアプリケーション以外にもモバイルであったりバッチであったり、その他のシェルなどから複数の窓口を設けたい場合というのは、基本的にはエンドポイントというのが必要になりますので、Root Handlersを使うことになります。サーバーアクションだとエンドポイントそもそも公開できないものになるのでそうした複数のクライアントから呼び出されるものに関してはRoot Handlersを使うのが良いかと思いますまたアプリケーションの依存度を下げるという観点でもサーバーアクションよりはRoot Handlersの方が優れていると思いますRootHandlersっていうのはAPIエンドポイントになるのでそれは他のいろんなシステムで共通のプロトコルだと思います対してServerActionsっていうのは結構Next.js独自の仕様になっているので例えばこの処理を他のシステムに移植したいだとかっていうのがあった場合にサーバーアクションズだと移行コストっていうのが大きくなってしまうかなっていう懸念もあります続いてそもそも何をするのかっていう観点でいくと データの更新とか であればサーバーアクションを使うのが比較的工数を抑えられる 選択肢だと考えていて これも先ほどのから呼ばれるのかっていうのが アプリケーション内に限定された前提なんですけども そうした場合はサーバーアクションを使うことで 最初の問いでも説明 してるとおり ユーザーアクションステートとの相性 リクエストの 制御っていうのはかなりルートハンドラーズと比較して簡易簡潔 に可読性高くできるので そうした最小化にしてデータの更新に関してはサーバーアクションを使うのが良いと思っています。また型の変換なども意識しなくていいので、共通の型を使い回せるといった面でも良いかと思いますまたセキュリティという観点からしてもサーバーアクションズの方がNext.jsが自動でCSRFの対策などもしてくれるので安全かなと思いますRootHandlersだとリクエストを自前で構築する必要が出てきますので自前でCourseの設定であったりCSRFであったり認証、オーストラリジェンションヘッダーなどを付与する必要があって場合によってはそれが漏れてたりしてセキュリティホールとなってしまうケースもあるのでそういったリスクを極力下げるという意味でもサーバーアクションでは良いかと思います

> **レビューコメント:**
>
> **CRUD 操作の違い** — Route Handlers との実装コスト差を的確に説明できています。合格です。
>
> **使い分け** — 非常に多角的な分析ができていて素晴らしいです。以下の観点がすべて押さえられています:
> - **誰が呼ぶか**（外部クライアント → Route Handlers、アプリ内 → Server Actions）
> - **移植性**（Server Actions は Next.js 独自仕様、Route Handlers は標準的な HTTP API）
> - **型安全性**（Server Actions は TypeScript の型をクライアント・サーバー間で共有可能）
> - **セキュリティ**（Server Actions は CSRF 対策が自動、Route Handlers は自前で対応）
>
> 補足として1点だけ: 「サーバーアクションだとエンドポイントそもそも公開できない」という記述がありますが、これは Q5 のセキュリティの問いとも関連します。実は Server Actions も裏側では HTTP エンドポイントが生成されています。「公開 API として設計されていない」という表現の方がより正確です。この点は Q5 で深掘りしてみてください。
>
> また、もう一つの使い分け観点として — **データの読み取り（GET）** には Server Actions は使えません（Server Actions は POST のみ）。データ取得には Server Component で直接取得するか、Route Handlers（GET）を使うことになります。この点も押さえておくと完全です。

## Q5: Server Actions のセキュリティ

**Server Actions が裏側で生成する HTTP エンドポイント:**
Server Actions は、Route Handlers と同じように公開 HTTP のエンドポイントを持っています。基本的には POST リクエストのみ受け付けています。そちらのエンドポイントは、Server Actions が呼び出されるオリジンがリソース名になっていて、そのリソースの中でアクション ID に紐づいて内部で分岐しているという構造になっています。Route-handlers は基本的にはリソースのパスとメソッドで 一意に API エンドポイントが定まるんですけど、Server Actions の場合は、基本的にエンドポイント自体は関数が実行されるオリジンに集約されていて、その内部でアクション ID によって分岐するというイメージになっています。


**自アプリのフォーム以外から呼び出せるか:**
やろうと思えば呼び出せます。上記の問いですでにサーバーアクションが公開エンドポイントとして公開されているという話をして、さらにアクションIDをヘッダーに付与すれば理論上呼び出せます。アクションID自体もクライアントにバンドルされたものの中に混在している状態なので、やろうと思えばそこから抽出してAPIを叩くことができます。ただ、呼び出せますけども、内部的にCSRFの対策がされていて、例えば自分アプリじゃなくて他のアプリのフォームから呼び出そうとすると強制的にオリジンとホストがミスマッチするのでその時点で拒否されます。


**バリデーションや認証チェックの重要性:**
上記で説明した通り、Server Actions は公開エンドポイントになるので、極限的には誰でも呼び出せるという風になっているので、適切にバリデーションしたり、認証チェックを怠ってしまうと、ハックされてしまう危険性があるので、バリデーションと認証チェックは重要かなと思います。


**「フォームからしか呼ばれないから入力チェックは不要」は正しいか:**
フォームから呼ばれないというのがまず違くて、普通に多分変えるコマンドでシェルから実行することもできると思います。その際にリクエストボディに不適切な値を入れたりすることもできると思うので、そういうことを加味すると、入力チェックはサーバーアクションズ側でも必要かなというふうに思っています。

> **レビューコメント:**
>
> **エンドポイントの構造** — Route Handlers との対比が的確です。「パス+メソッドで一意に定まる Route Handlers」vs「オリジンに集約されアクション ID で分岐する Server Actions」という整理は良い理解です。合格です。
>
> **外部からの呼び出し** — 合格です。アクション ID がクライアントバンドルに含まれているため抽出可能という点、CSRF 対策によるオリジンチェックの点、両方正確に押さえられています。
>
> 1点補足: CSRF 対策は「**別オリジン**からの呼び出し」を防ぎますが、**同一オリジン**からの curl や DevTools での呼び出しは防げません。つまり、ログインしているユーザーが DevTools からアクション ID を使って不正なペイロードを送ることは可能です。これがバリデーションの重要性に直結します。
>
> **バリデーション・認証チェック** — 結論は正しいです。より強く言うと、Server Actions は **Route Handlers と同じレベルのセキュリティ意識** が必要です。「サーバーサイドのコードだから安全」ではなく「公開エンドポイントだから信頼できない入力が来る前提で書く」という原則です。具体的には:
> - **入力バリデーション**: zod 等で必ずサーバー側でも検証する（クライアント側のバリデーションはUXのためであり、セキュリティ対策ではない）
> - **認証・認可チェック**: セッションの確認、操作権限の確認を Action 内で行う
> - **レート制限**: 必要に応じてリクエスト頻度を制限する
>
> **「入力チェック不要」への反論** — 正しい結論に到達しています。curl での呼び出しに言及している点も良いです。
