# 演習3-1: 言語化演習 回答

## Q1: Route Handlers と Server Actions の使い分け

**どんなクライアントからのリクエストを想定しているか:**
サーバーアクションズはアプリケーションの内部のみから呼ばれることを想定しています。ルートハンドラーズはアプリケーションの内部に加えて、外部のクライアントからのリクエストも想定しています。例えば、定期的にスケジューリングされているバッジからのリクエストであったり、モバイルあるいはシェルなどから呼ばれることも想定しています。

**データの取得（Read）と変更（Write）での使い分け:**
データ取得に関しては、そもそもルートハンドラーズであったりサーバーアクションを使わずにサーバーコンポーネントで完結するケースが多いということは念頭に置いておいて、その上でルートハンドラーズかサーバーアクションと言うと、基本的にルートハンドラーズを使うのが好ましいかなと思います。理由としては、Route Handlersの場合はURLとして公開されているものなので、HTTPレイヤーでそれぞれキャッシュなどの適用ができるというところと、データ取得をするクライアントがアプリケーション以外でも他のモバイルであったり他のバッジなどのシステムからも同じ共通エンドポイントを利用できるという面でルートハンドラーズを使うのが好ましいかなというふうに考えます。変更に関してはRoute HandlersよりもServer Actionsの方が好ましいという風に考えています。理由としては、まずServer Actionsの場合は、アプリケーションとの相性、アプリケーションのロジックコードとの相性が良いので、例えばフォームで入力書をサーバーに転送するみたいなものだと、転送した後にエラーハンドリングをしたり、ローリングをしたり、その後画面をリフレッシュしたりっていうような、その処理にちなみに画面が制御する必要があるので、そうした画面制御との相性っていうのを考えると、サーバーアクションズの方がRoute Handlersよりもやりやすいです。また型の統一感も、言ってしまえばサーバーアクションズというのは関数として登録できるものになるので、プロダクションコードの型というのもそのまま統一しやすいという利点があります。またセキュリティの点からも外部に公開するものではないものになるので、そもそものアタックサーフェイスを縮小することにもなりますし、CSRF対策もNext.jsが組み込みでやってくれるので、セキュリティ的にもローコストで実装できるという強みがあるかなと思います。ただ、こうした更新の中でも外部に公開したい、例えばWebhookなどで連動させたいみたいなケースに関しては、Route Handlersを使う必要が出てきます。

**URL を持つかどうかの違いが設計に与える影響:**
URLを持つ場合、curlやPostmanなどを外部から呼び出すことができ、さらにOpenAPIなどでドキュメンテーション化することができます。また、CDNやブラウザでキャッシュすることができます。
URLなしの利点に関しては、基本的にエンドポイントが外部に公開されないので、バックドアがそもそも存在しないということで、セキュリティ的に好ましいのと、あとはNext.jsが自動的にCSRF対策を自動で行う点ですね。

## Q2: Route Handlers のキャッシュ挙動

**GET の結果がキャッシュされる条件:**
まず、GET メソッドであること、Request オブジェクトを使っていないこと、cookies() や headers() などの動的関数を使っていないこと、同じファイルに GET 以外のメソッドがないこと、そうして静的な関数として定義されているというのが条件になります。

**キャッシュされないようにする方法:**
上記で説明している GET の結果がキャッシュされる条件に当てはまらないことをすれば良いと思います。まず、リクエストオブジェクトを使う。クッキーズやヘッダーズを使う。またファイルの先頭に `export const dynamic = "force-dynamic" `の宣言をする。同じファイルに POST/PUT/DELETE を定義する。

**Server Component の fetch() のキャッシュとの違い:**
まずキャッシュの単位という観点でいうと、サーバーコンポーネントのフェッチに関しては、そのフェッチを実行するリクエストごとにキャッシュが設定されます。ざっくり言うと、指定したリクエストURLごとかなと思います。対してルートハンドラーズの場合は、ルートハンドラーズ自体、つまりGETのエンドポイントを○○キャッシュとして扱われます。また再検証の仕組みですが、フェッチの場合はオプションで再検証の間隔などを指定します。対してroot-handlersの場合はopt-inとして、再検証の間隔を設定します。いずれも同じようなこと、ISRのような挙動を実現できるかと思います。

## Q3: NextRequest と Web API の Request

**NextRequest で追加されている便利な機能:**
URLオブジェクトを取得するプロパティであるnextUrlであったり、クッキーズの読み書きが簡単にできるcookiesというメソッドなどが組み込まれています。URLを操作したり、クッキーを扱ったりするための便利な機能が追加されています。また、デプロイ環境にもよるんですけども、ジオグラフィーやIPアドレスなどを取得するプロパティも存在します。

**どちらを使うべきかの判断基準:**
Next.jsに依存させたくないのであればWeb標準のリクエストを用いるべきかなと思います。ただ今後もNext.jsでずっとやっていくっていう中でURLであったりクッキーなどを便利に扱いたい場合はNext.jsリクエストを使えば良いかと思います。

## Q4: Edge Runtime と Node Runtime

**それぞれの実行環境はどこで動くか:**
Edge Runtimeは、できるだけクライアントの近くのエッジロケーションから配置されているエッジサーバーで動きます。対して、Node Runtimeは、オリジンが存在するサーバーで動きます。

**Edge Runtime で使えない Node.js API:**
Edge Runtimeが使えないNode.jsのAPIとしては、ファイルシステムであったりchild_processなどといった重めの処理を扱うライブラリなどは利用できないです。理由としては、Edge Runtimeはそういった重めの処理に向けての環境ではなく、軽量な処理に備えた環境であるためです。

**Edge Runtime が向いている API・向いていない API:**
Edge Runtimeが向いているAPIというのは、基本的には軽量な処理になります。具体的に言うと、リクエストのルーティングであったり、印象認可のガードレールを敷いたり、あるいは地理的なルーティングといったちょっとの処理で完結できるようなAPIになります。逆に向いていないAPIは基本的に重めの処理で、例えばDatabaseでアクセスしたりだとか、マシンのファイルシステムでファイルを書き換えたりだとか、というような重い処理は向いていないかなと思います。

**コールドスタートの速度差の理由:**
Edgeの方がNodeよりもコールドスタートは早いです。理由としては、Edgeの実行環境がNodeと比べて軽量に動く、Web純粋のAPIなどを備えていれば良いので、そもそも準備にかかる時間っていうのがソースコードをロードしたりだとかっていうのがあるんですけど、そういった下準備の処理がNode Runtimeほど時間がかからないためです。
