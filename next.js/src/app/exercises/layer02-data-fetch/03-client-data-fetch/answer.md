## Q2: stale-while-revalidate 戦略

**ユーザーがページにアクセスしたとき、何が起きるか:**
ユーザーがページにアクセスした場合は、キャッシュが存在する場合は即座にキャッシュをコンテンツとして表示します。その裏でSWRは最新のコンテンツを取得、つまり再検証をし、その再検証が完了した後に画面のコンテンツを最新化します。


**なぜこの戦略がUXの向上につながるのか:**
常にユーザーのアクセスに対して高いスピードでコンテンツを表示することが可能だからです。仮に基本キャッシュを返して一定期間経ったら最新化したものを返すという風になると、どこかのタイミングでコンテンツの表示が遅くなってしまいローディングを挟むが、stale-while-revalidate ではそういったことは起こらず、仮にそのコンテンツを最新化する場合だったとしても、一旦はキャッシュを表示するということで、安定して高いパフォーマンスでコンテンツを表示することができます。


**HTTP の Cache-Control ヘッダーとの関連:**
SWRの内部で実行されるFetcherのパフォーマンスにHTTPのCache-Control Headerが関連してきます。SWRでCacheを再検証するときにFetcher内部でAPIを実行すると思うんですけど、そのAPIの実行結果のレスポンスにCache-Control HeaderでレスポンスをHTTPキャッシュするという設定があった場合にSWRで再検証する際にフェッチャーが実行されるんですが、そのときに実際にAPIを実行してオリジンに問い合わせるのではなく、HTTPキャッシュの値を再利用することができるので、結果としてSWRの再検証のパフォーマンスが上がります。
ちなみにHTTPの仕様でもCache-Controlのディレクティブとしてstale-while-revalidateというものがあり、SWRのライブラリの由来はここから来ています。つまりアプリケーションとHTTPこの2つのレイヤーにおいて同じstale-while-revalidateというキャッシュ戦略が有効であるということがわかります。

注意点として、SWRのキャッシュとHTTPキャッシュは別のもので、SWRのキャッシュはブラウザのメモリに保存されている、アプリケーション内で管理されているキャッシュになります。SWRの再検証などの制御は基本的にこのアプリケーション内のキャッシュによって制御されるので、HTTPキャッシュがあるからといってSWRの制御に絡むことはありません。

## Q3: Client データフェッチで必要になるもの

**なぜ Provider が必要なのか:**
アプリ全体でコンポーネントすべてに対して同じ共有キャッシュを同期する、そのスコープを定めるためにプロバイダーが必要になります。プロバイダーでラップした配下のコンポーネント群は、TanStack Query Client でキャッシュされたものを共有することになるので、コンポーネント間で同じキャッシュを同期することができるようになります。


**App Router での Provider の配置場所と理由:**
プロバイダーはトップレベルに配置します。具体的に言うと、layout.tsx でプロバイダーを設置して、アプリ全体をプロバイダーの監視下に置くことで、アプリ全体で統一したキャッシュを参照することができるようになります。ただし、layout.tsx 自体はサーバーコンポーネントになるため、プロバイダーはクライアントコンポーネント。React のコンテキストを使用するのでクライアントコンポーネントにしなければいけないため、layout.tsx でプロバイダーをインポートする形で適用します。

## Q4: React `use` API の特徴

**`use` は何を受け取って、何を返すか（Promise の場合）:**
Promiseを受け取り、値が解決されない場合はPromiseをそのままスローし、Promiseが解決された場合は解決された値を返します。


**`use` は何を受け取って、何を返すか（Context の場合）:**
コンテキストを受け取り、コンテキストで管理している値を返します。


**`use(promise)` 使用時のコンポーネントの振る舞い（Suspense との関係）:**
受け取ったプロミスが未解決の場合は、サスペンスに対してプロミスをスローします。それを受け取ったサスペンスはフォールバックで指定したコンポーネントを表示します。サスペンス内でプロミスが解決されると、それをトリガーにラップしたコンポーネントを再レンダリングします。その際に use は解決された値を返します。


**SWR / TanStack Query と比べて `use` API だけではできないこと:**
Use APIでは取得したデータのキャッシュを扱うことができません。毎回データをオリジンから取得する必要があります。対してSWR・TanStack Queryはキャッシュ戦略を取り入れており、一定期間はメモリに保存した値を再利用するようにし、かつデータ取得はバックグラウンドで並行して行い、最新化されたタイミングで画面の表示を更新する、といったUXの向上が見込めます。データ取得時のハンドリング制御も充実しており、isErrorだったりisLoadingだったりっていうのがデフォルトで組み込まれているので、そうしたデータ取得の状態に応じた制御っていうのがしやすくなっています。また、定期的に再取得するポーリングであったり、フォーカス時の再取得であったり、楽観的な更新などもuseにはない機能になっています。


**Server Component → Client Component に Promise を渡すパターンが有効な場面:**
データ取得に時間がかかる場合で、かつ一旦データ取得以外の画面表示を即座に行いたい場合に、プロミスをクライアントコンポーネントに渡すという設計がUXの向上につながると思います。サーバーコンポーネントの中で、await.fetchしてデータを完全に取得して、HTMLを返すというやり方だと、そのHTMLが構成されるまで、ユーザーは待機することになります。その間、画面には何も表示されません。それだと、UX的には待ち時間が長くなってしまうので、不利なんですけど、これを一旦プロミスの生成だけサーバー側でやって、とりあえず一旦クライアントに返して、そのデータ取得とは関係ない部分の外側をすぐ表示するっていう風にして、データ取得の部分をローディングっていう風にしてあげると、ユーザーからしたらとりあえず画面が見れる状態になるので、UXはある程度向上するかなと思います。
